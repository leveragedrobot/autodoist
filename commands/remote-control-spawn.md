# Remote Control Spawn

Spawn a Claude Code session on a remote machine to execute tasks remotely. Use this when a task requires direct access to a remote server's filesystem, services, or environment.

## When to Use

Use remote-control-spawn instead of inline SSH commands when:
- The task requires **multiple commands** on the remote machine (not just one `ssh` call)
- The task needs **project context** from the remote filesystem (reading code, running tests, deploying)
- The task involves **interactive troubleshooting** (diagnose → fix → verify cycles)
- The task would burn excessive tokens piping SSH output back and forth

**Don't use** for single-command tasks like `ssh host 'systemctl restart nginx'` — just run those inline.

## Step 1: Validate Remote Target

Before spawning, confirm the remote machine is reachable and Claude Code is available:

```bash
# Check SSH connectivity (timeout 5s)
ssh -o ConnectTimeout=5 -o BatchMode=yes USER@HOST 'echo ok' 2>&1

# Check Claude Code is installed on the remote
ssh -o ConnectTimeout=5 USER@HOST 'which claude 2>/dev/null || echo NOT_FOUND'
```

| Result | Action |
|--------|--------|
| SSH fails | Block task — log "Remote server unreachable" |
| Claude not found | Fall back to SSH-only execution (limited) |
| Both pass | Proceed with remote spawn |

## Step 2: Prepare Task Payload

Build the task instructions to send to the remote Claude session. The payload should be self-contained — the remote session has no context from the local autodoist run.

**Payload template:**
```
Task: [task title from Todoist]
Project: [project name]
Working directory: [remote project path]
Instructions:
[step-by-step instructions, either from task description or generated by autodoist]

When done, output a summary in this format:
---RESULT---
status: [completed|failed|partial]
summary: [what was done]
files_changed: [list of files modified, if any]
errors: [any errors encountered]
---END---
```

**Important:** Include enough context for the remote session to work independently. Specify:
- The exact working directory on the remote machine
- Any environment variables or config needed
- Success criteria so the remote session knows when it's done

## Step 3: Spawn Remote Session

Launch Claude Code on the remote machine with the task payload:

```bash
# Spawn remote Claude session with task
ssh -o ConnectTimeout=10 USER@HOST "cd PROJECT_DIR && claude -p '$(cat <<'TASK'
[task payload from Step 2]
TASK
)'" 2>&1
```

**For long-running tasks**, use a detached approach:

```bash
# Write task to a temp file on the remote
ssh USER@HOST "cat > /tmp/claude-task-$(date +%s).md << 'TASK'
[task payload]
TASK"

# Execute Claude with the task file
ssh USER@HOST "cd PROJECT_DIR && claude -p \"\$(cat /tmp/claude-task-*.md)\"" 2>&1
```

### Timeout Handling

Set a timeout proportional to the task complexity:

| Task Type | Timeout | Example |
|-----------|---------|---------|
| Quick fix / config change | 2 min | Restart service, update env var |
| Code change + test | 10 min | Implement feature, run tests |
| Deploy pipeline | 15 min | Build, test, deploy, verify |
| Complex investigation | 20 min | Debug production issue |

```bash
# With timeout
timeout 600 ssh USER@HOST "cd PROJECT_DIR && claude -p '[task]'" 2>&1
```

## Step 4: Collect Results

Parse the remote session output for the `---RESULT---` block:

```bash
# The output from Step 3 contains the result
# Parse for the structured result block
```

**Result handling:**

| Status | Action |
|--------|--------|
| `completed` | Log summary as Todoist comment, mark task complete |
| `failed` | Log error as Todoist comment, keep task open, add to Blocked |
| `partial` | Log progress as Todoist comment, create subtasks for remaining work |
| No result block | Treat as failed — remote session may have crashed or timed out |

**Log the result:**
```bash
td comment add --task-id <id> "Remote execution on HOST:
Status: [status]
Summary: [summary]
Files changed: [files]"
```

## Step 5: Error Recovery

### Connection dropped mid-execution
If SSH drops during execution, the remote Claude session may still be running:

```bash
# Check if Claude is still running on remote
ssh USER@HOST 'pgrep -f "claude" && echo RUNNING || echo STOPPED'
```

- If **RUNNING**: wait and reconnect to check output
- If **STOPPED**: check for partial results in the remote project (git log, recent file changes)

### Remote session failed
If the remote Claude session exits with an error:

1. Check the error output for actionable information
2. If it's a missing dependency or permission issue, log the blocker
3. Do NOT retry automatically — remote failures often indicate environment issues

### Timeout exceeded
If the task exceeds its timeout:

1. Kill the remote process: `ssh USER@HOST 'pkill -f "claude -p"'`
2. Check for partial work on the remote (git status, recent commits)
3. Log partial progress and create follow-up subtask

## Remote Hosts Configuration

Define your remote hosts and their capabilities. Autodoist uses this to route tasks to the right machine.

| Host Alias | Connection | Claude Available | Use For |
|-----------|------------|-----------------|---------|
| `prod` | `user@prod.example.com` | Yes | Deploy verification, log analysis |
| `staging` | `user@staging.example.com` | Yes | Full deploy + test cycles |
| `dev` | `user@dev.example.com` | Yes | Code tasks, builds, tests |

Customize this table with your own remote hosts.

### Host-to-Project Mapping

Map projects to their remote hosts so autodoist can route automatically:

| Project | Remote Host | Project Path |
|---------|------------|-------------|
| Backend API | `dev` | `/home/user/api` |
| Frontend | `dev` | `/home/user/web` |
| Infrastructure | `staging` | `/home/user/infra` |

## Integration with Autodoist

### Signal 3 Additions

Add these to the Skill Mapping table in `autodoist.md`:

| Task Pattern | Skill Chain | Execution Method |
|-------------|------------|-----------------|
| "deploy" + remote project | `/remote-control-spawn` → `/deploy` | Spawn on remote host → deploy → verify |
| "fix" + server/backend | `/remote-control-spawn` → `/code-task` | Spawn on remote → implement fix → test |
| "check logs", "investigate" + server | `/remote-control-spawn` | Spawn on remote → analyze → report |
| "restart", "configure" + service | `/remote-control-spawn` | Spawn on remote → execute → verify |

### Chain Examples

**Remote Code → Test chain:**
```
Chain name: Remote fix and test
Trigger: coding task + remote project
Steps:
  1. /remote-control-spawn  → spawns Claude on remote host
     HANDOFF: task payload with code instructions
  2. Remote Claude executes  → /code-task → /run-tests (on remote)
     HANDOFF: result block (status, summary, files changed)
  3. Parse result            → check status
     ON FAIL: log blocker, stop chain
  4. Log + complete          → comment on task, mark done
```

**Remote Deploy chain:**
```
Chain name: Remote deploy
Trigger: "deploy" + remote project
Steps:
  1. /remote-control-spawn  → spawns Claude on staging/prod
     HANDOFF: deploy instructions
  2. Remote Claude executes  → build, test, deploy
     HANDOFF: result block with deploy URL
  3. GATE                    → show deploy result, ask user to verify
  4. Log + complete          → comment with deploy URL, mark done
```

**Remote Investigation chain:**
```
Chain name: Remote investigate
Trigger: "check", "investigate", "debug" + server/service
Steps:
  1. /remote-control-spawn  → spawns Claude on target host
     HANDOFF: investigation instructions
  2. Remote Claude executes  → analyze logs, check services, diagnose
     HANDOFF: result block with findings
  3. Report findings         → present to user with recommended actions
  4. GATE                    → user decides: fix now, create task, or dismiss
```
